# Безопасность HubX

Здесь собран разбор найденных проблем и того, как они закрыты. Каждую уязвимость сначала подтверждали рабочим PoC, потом исправляли и проверяли заново. Область проверки: `server/` (backend), `ejabberd/`, `web/` (клиент), `examples/`.

## Сводка

| # | Проблема | Уровень | CWE | Статус |
|---|-----------|----------|-----|--------|
| 1 | Удаление любого аккаунта без авторизации (`DELETE /api/users/:u`) | Critical | CWE-862 | исправлено |
| 2 | Перечисление всех пользователей без авторизации (`GET /api/users`) | High | CWE-200 / 862 | исправлено |
| 3 | Открытый CORS `Access-Control-Allow-Origin: *` | Medium | CWE-942 | исправлено |
| 4 | Нет ограничения частоты на регистрации | Medium | CWE-770 | исправлено |
| 5 | Утечка версии (`X-Powered-By`) и текстов ошибок ejabberd | Low | CWE-200 | исправлено |
| 6 | Секреты по умолчанию в коде | Low | CWE-798 | смягчено |

`npm audit` по продакшен-зависимостям не показывает уязвимостей. Клиент выводит сообщения только как текст, React их экранирует, `dangerouslySetInnerHTML` не используется, поэтому XSS через содержимое сообщений не проходит. Вложения дополнительно фильтруются по схеме URL: ссылки с `javascript:` и `data:` отбрасываются.

## Что было и что стало

**1. Удаление аккаунта без авторизации.** `DELETE /api/users/:username` работал без проверки прав. До фикса любой запрос стирал чужой аккаунт:

```bash
curl -X DELETE http://localhost:4000/api/users/victim   # {"ok":true}
```

Теперь эндпоинт закрыт middleware `requireAdmin` (валидный HMAC-токен), удалить свой аккаунт нельзя:

```bash
curl -i -X DELETE http://localhost:4000/api/users/someuser   # HTTP 401
```

**2. Перечисление пользователей.** `GET /api/users` отдавал список всех JID без авторизации. Теперь требует админ-токен, без него отвечает `401`.

**3. Открытый CORS.** `cors()` без опций отражал любой Origin. Добавлен аллоулист (`ALLOWED_ORIGINS`, по умолчанию `localhost:5173`). Запрос с чужого origin не получает заголовок `Access-Control-Allow-Origin`.

**4. Частота запросов.** Подключён `express-rate-limit`: регистрация 10 запросов за 15 минут, вход в админку 20 запросов за 15 минут с одного IP.

**5. Утечка информации.** Отключён `x-powered-by`, добавлен `helmet` (CSP, `X-Content-Type-Options: nosniff`, готовность к HSTS). Клиент получает обобщённые тексты ошибок, детали ejabberd остаются в логах сервера.

**6. Секреты по умолчанию.** Пароли вынесены в переменные окружения (`ADMIN_PASS`, `ADMIN_API_SECRET`). Если секрет не задан, сервер печатает предупреждение и генерирует случайный HMAC-ключ через `crypto.randomBytes`. Минимальная длина пароля при регистрации поднята до 8 символов. В продакшене сервер не стартует с встроенными значениями.

## Аутентификация админ-API

Открытое админ-API заменено на токены:

1. `POST /api/admin/login {username, password}` проверяет пароль администратора в ejabberd (`check_password`) и убеждается, что пользователь входит в список админов.
2. При успехе выдаётся HMAC-SHA-256-токен на 30 минут: `base64url(payload).base64url(HMAC(secret, payload))`.
3. `GET /api/users` и `DELETE /api/users/:u` требуют заголовок `Authorization: Bearer <token>`. Подпись сверяется в постоянном времени (`crypto.timingSafeEqual`), просроченные и подменённые токены отвергаются.

```bash
TOK=$(curl -s -X POST -d '{"username":"admin","password":"ВашПароль"}' \
      -H 'Content-Type: application/json' http://localhost:4000/api/admin/login | jq -r .token)
curl -s -H "Authorization: Bearer $TOK"  http://localhost:4000/api/users    # 200
curl -s -H "Authorization: Bearer ${TOK}x" http://localhost:4000/api/users  # 401, подпись неверна
curl -s -X POST -d '{"username":"user","password":"userpass"}' \
      -H 'Content-Type: application/json' http://localhost:4000/api/admin/login  # 403, не админ
```

Клиент получает токен сам: после XMPP-входа администратора приложение обменивает учётные данные на токен и держит его в памяти, не в localStorage.

## Секретные чаты

Личная переписка в секретном режиме шифруется на стороне клиента (ECDH P-256 + AES-GCM). Сервер видит и хранит только шифртекст. Ключ собеседника подписывается долговременным ключом-удостоверением (ECDSA), поэтому подмену видно сразу, а отпечаток можно сверить вживую. Идентификатор сообщения и таймер самоуничтожения лежат внутри шифртекста, так что сервер не может незаметно их подменить. Вложения шифруются тем же ключом и грузятся как непрозрачный блоб.

## Защита данных в админ-панели

IP входа и геолокация это персональные данные. Они доступны только администратору по токену, не отдаются обычным пользователям, не попадают в репозиторий (`server/data/` в `.gitignore`) и имеют срок хранения. При удалении аккаунта его сохранённый IP стирается.

## Доступ к API сервера и заголовки

HTTP-API ejabberd отвечает только с loopback или под аутентификацией администратора, анонимные вызовы отклоняются. В прод-конфигурации порт ejabberd наружу не выходит: Caddy проксирует только `/ws` и `/upload`, а API живёт внутри docker-сети. Список доступных команд сужен (`api_permissions`, секция `admin access`) до того набора, которым реально пользуется backend: `register`, `unregister`, `check_password`, `check_account`, `registered_users`, `connected_users`, `connected_users_info`, `get_offline_count`, `get_room_affiliations`, `set_room_affiliation`, `status`. Остальные команды через API недоступны, даже под админом.

В облачном развёртывании Caddy добавляет заголовки безопасности:

- `Strict-Transport-Security` со сроком год для всего домена.
- `Content-Security-Policy`, настроенный под реальные источники приложения: скрипты только свои (`script-src 'self'`), стили свои плюс Google Fonts, шрифты с `fonts.gstatic.com`, картинки из `data:`/`blob:` (аватары и вложения), соединения к своему WebSocket (`connect-src 'self' wss://<домен>`), service worker свой. `object-src` и `frame-ancestors` запрещены.
