# HubX: развёртывание в облаке

Есть два способа выложить мессенджер в интернет по HTTPS.

- **Способ A**: постоянный сервер на бесплатной облачной VM. Работает круглосуточно, свой компьютер можно выключить. Это основной вариант.
- **Способ B**: туннель с ноутбука. Подходит для разового показа и работает только пока включён ваш ПК.

---

## Способ A. Постоянный сервер: Oracle Cloud Always Free + DuckDNS

Весь стек (ejabberd, backend, бот, веб-клиент) работает в Docker на удалённой VM. Caddy принимает HTTPS на порту 443 и сам получает сертификат Let's Encrypt.

```
браузер --HTTPS--> Caddy(:443)  /          -> web (статика в образе)
                                /api/*     -> backend:4000
                                /ws,/upload -> ejabberd:5280
```

### Шаг 1. Создайте VM в Oracle Cloud

Зарегистрируйтесь на [oracle.com/cloud/free](https://www.oracle.com/cloud/free/) и создайте инстанс уровня Always Free с Ubuntu 22.04. Архитектура ARM Ampere подходит.

Откройте входящие порты TCP 80 и 443 в двух местах:

1. В Security List вашей VCN (в веб-консоли Oracle).
2. В файрволе самой VM. Образы Ubuntu у Oracle блокируют эти порты по умолчанию:
   ```bash
   sudo iptables -I INPUT -p tcp --dport 80 -j ACCEPT
   sudo iptables -I INPUT -p tcp --dport 443 -j ACCEPT
   sudo netfilter-persistent save
   ```

### Шаг 2. Заведите поддомен на DuckDNS

Зайдите на [duckdns.org](https://www.duckdns.org) через аккаунт GitHub или Google. Создайте поддомен и впишите в него публичный IP вашей VM. Пример: `hubx-mesh-app.duckdns.org`.

### Шаг 3. Разверните стек на VM

```bash
sudo apt update && sudo apt install -y git
git clone https://github.com/Danyashvnc/hubx.git && cd hubx
bash deploy/vps-setup.sh
```

При первом запуске скрипт создаёт файл `.env` и останавливается, чтобы вы его заполнили. Откройте `.env` и задайте три переменные: `DOMAIN` (ваш поддомен DuckDNS), `ADMIN_PASS` и `BOT_PASS`. Затем запустите скрипт второй раз:

```bash
nano .env
bash deploy/vps-setup.sh
```

Второй запуск ставит Docker, генерирует секрет и сертификат, собирает образы, поднимает контейнеры (ejabberd + backend + бот + Caddy), создаёт аккаунт `admin` и печатает адрес. Caddy получает сертификат Let's Encrypt при первом обращении к домену, на это уходит около 30 секунд.

### Шаг 4. Проверьте результат

Откройте `https://ваш-поддомен.duckdns.org`. Сервер работает круглосуточно. Единственный созданный аккаунт: `admin` с паролем из `.env`. Остальные пользователи регистрируются сами через приложение.

Управление (на VM, в каталоге `hubx`):

```bash
docker compose -f docker-compose.prod.yml logs -f      # логи
docker compose -f docker-compose.prod.yml restart      # перезапуск
docker compose -f docker-compose.prod.yml down         # остановка
```

## Способ B. Туннель с ноутбука (Cloudflare Tunnel)

Cloudflare Tunnel выводит локальный стек наружу по временному адресу вида `https://имя.trycloudflare.com`. VPS, белый IP, домен и открытые порты не нужны. Ограничение: всё работает, только пока включён ваш компьютер.

Запуск:

```powershell
# 1. Базовый стек
docker compose up -d                 # ejabberd
cd server; npm start                 # backend :4000 (отдельное окно)
cd ..\web; npm run build             # сборка веб-клиента в web/dist

# 2. Прокси и туннель
cd ..
docker compose -f docker-compose.cloud.yml up -d

# 3. Публичный адрес
docker logs hubx-tunnel | Select-String "trycloudflare.com"
```

Последняя команда печатает адрес стенда. Учтите: при каждом перезапуске туннеля адрес меняется. Постоянный адрес даёт только именованный туннель Cloudflare, для него нужен аккаунт Cloudflare и свой домен.

Остановить: `docker compose -f docker-compose.cloud.yml down`.

### Вложения через туннель

ejabberd по умолчанию выдаёт ссылки на загрузку файлов с адресом `localhost`, снаружи они не открываются. Пропишите публичный адрес в `ejabberd/conf/ejabberd.yml`, секция `mod_http_upload`:

```yaml
put_url: "https://<ваш-адрес>.trycloudflare.com/upload"
get_url: "https://<ваш-адрес>.trycloudflare.com/upload"
```

Затем перезапустите ejabberd:

```bash
docker compose restart ejabberd
```

## Замечание про домены

Веб-клиент определяет адрес сервера сам: если страница открыта не с `localhost`, клиент подключается к `wss://тот-же-хост/ws` и `https://тот-же-хост/api` (логика в `web/src/config.ts`). Пересобирать клиент под каждый домен не нужно. Локальная разработка через `npm run dev` продолжает работать как раньше.
